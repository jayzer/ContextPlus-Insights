# ContextPlus-Insights
## Part #1 - Lessons Going to Market with Reeve Tech LLC: ContextPlus Memory 

A Co-Founder's note.  A blog-style reflection of the process of recruiting beta testers and winning over the first customers, executing that go-to-market strategy.

So we've done the groundwork.  Our team is ready to go.  The workflow has been set up for design partners to start.  What am I talking about?

Memory.
Context.
These could be the words of the decade, the "other problem" when it comes to AI.  LLMs can become stronger in their comprehension and reasoning capabilities.  They can code like crazy.  They can have access to a bunch of tools.  But one problem is always there.  I discussed this in a recent five-minute investor pitch video.  The problem is that AIs always forget what's going on.  They are short-sighted in their memories.   You'll never really have a good coworker in your development team when it comes to using an AI.  You come back to a problem later, start a new Claude Code session, and voilà: 

Assistant: How can I help you?  
User: I want to go to the next phase and upgrade the user interface, add a way for users to manipulate data in 3D and zoom in to specific sections, spread it out into 2D. 
Assistant: [Makes some random thing, changes a bunch of code that didn't need to be changed.]

The problem? I had spent hours and days discussing so many details about the project, the standards, the structure, the documentation.  Describing the peculiarities about my working style.  Now here Claude is changing our entire project after we had gone through lots of critical decisions already.  It forgot. 

Our solution? A supplementary and fully automated memory layer for professional coders working in their native coding environment that is conceptually situated between the project folder and the LLM.

**We Built It, We Think We've Solved a Real Problem**
After lots of work and iteration, it's done, it's deployed, our CEO/Chief Engineer has done his thing and it works.  The web page is up.  There is an early-bird signup, and there is a page for our design partners. 

**The Intersection of Marketing, Product Readiness, and Security Questions First**
Now is my time to shine as the head of marketing.  How do we get users to sign up here and use ContextPlus Claude Code with us?  One of the first questions we are getting is: how safe is my data? Do you have options to keep my data on premise? Can we segregate my data? Can I put it in a server where our company has the key? 

**Ok ok. For Now in Beta This is What We Have.  Is it Secure Enough To Attract Beta Testers/Development Partners?**
Those were the questions.  So I answer them.  Sure we can do those things.  At the moment, we're storing the data in industry-standard structures and enforcing least-permission access policies.  It's encrypted at rest and follows NIST post-quantum encryption standards.  We do not manually access or export user data on our side.  Oh one thing though, if you use LLMs they can see all of your data from the prompts and relevant project file being read after that, but it is our policy not to use it for training for you our Design Partners. 

So far, beta testers are few.  And the first few seemed to have paused.  Is it the cost of Claude, everyone already went over their token limit?  Do we need to deploy custom high security solutions for everyone before most people even do a hello world?  

These are the challenges of go-to-market, and starting with Development Partners on an unknown platform.  From our side we have solved something.  From the user side... I still need to  understand how users think about security and privacy. Should we expand the security documentation? 

Many questions to answer.  Treat this as day 1.  Development Partners, get people onboarded.  Go to market.  Reach everyone that may want memory, enthusiasts, heavy AI coders.  Vibe coders may actually be a good outreach group.   

-
Jay Cameron
Co-founder, Reeve Tech LLC: ContextPlus
